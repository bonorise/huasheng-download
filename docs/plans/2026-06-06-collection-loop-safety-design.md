# 收藏多轮下载安全修复设计

## 背景

收藏模式改为多轮提取后，每轮素材编号都会从 1 开始，导致后续轮次覆盖前一轮文件；同时，下载前写入去重集合会让下载失败的素材失去重试机会，取消收藏失败后也可能因为没有“新素材”而提前退出。`--limit` 也被错误地按每轮计算。

## 目标

- 任何已有视频文件都不能被覆盖。
- 收藏素材编号从输出目录现有最大编号的下一位开始，并在本次运行中持续递增。
- 下载失败的素材允许后续轮次重试。
- 下载成功但取消收藏失败的素材不重复下载，并在后续轮次继续尝试取消收藏。
- 收藏模式的 `--limit` 表示本次运行最多成功下载多少个素材。

## 方案比较

### 方案 A：扫描最大编号并使用排他写入

启动时扫描输出目录中的 `素材NN.mp4`，下一编号取最大值加 1。写文件使用排他模式，若目标文件已存在则重新分配下一编号后重试。

优点：历史文件顺序清晰；目录状态即使在运行中变化也不会覆盖。缺点：编号空洞不会复用。

### 方案 B：寻找最小空缺编号

每次寻找第一个不存在的 `素材NN.mp4`。

优点：编号紧凑。缺点：新批次会插入旧序列，文件时间和编号难以对应。

### 方案 C：文件名增加时间戳或随机后缀

完全绕开顺序编号冲突。

优点：冲突概率最低。缺点：破坏现有命名约定，不利于人工排序。

采用方案 A。

## 设计

### 文件编号与写入

新增纯函数解析收藏素材编号，并新增目录扫描函数计算下一编号。收藏模式在进入循环前计算 `nextMaterialNumber`，每轮提取后为新素材分配全局连续编号。

`downloadMaterial` 使用 `fs.writeFile(filePath, body, { flag: 'wx' })`。如果遇到 `EEXIST`，调用方增加编号并重试，确保即使运行期间出现同名文件也不会覆盖。

推荐模式保持原有按分镜编号方式，但写入同样使用排他模式；发现冲突时返回明确失败，避免覆盖旧文件。

### 多轮状态

运行期间维护两个 key 集合：

- `downloadedVideoKeys`：仅在下载成功后加入，避免重复下载。
- 待清理状态来自 `manifest.items`：`uncollectStatus !== 'uncollected'` 的已下载记录继续进入清理队列。

每轮顺序：

1. 打开收藏列表并提取可见素材。
2. 排除已经下载成功的 key。
3. 根据剩余总量限制截取本轮待下载素材。
4. 下载并记录成功 key。
5. 清理所有尚未成功取消收藏的已下载记录。
6. 若达到总量限制则退出；若列表为空则退出；若本轮没有新下载，但仍有清理失败项，则完成本轮清理后退出并保留失败记录，避免无限循环。

下载记录增加 `sourceKey`，便于从处理结果准确更新成功集合。

### `--limit`

收藏模式按本次运行成功下载数量累计。每轮传给提取器的限制为剩余额度；达到上限后停止继续提取和下载。推荐模式仍表示每个分镜最多数量。

### 测试

新增单元测试覆盖：

- 从多个文件名中计算收藏素材最大编号的下一位。
- 忽略不符合 `素材NN.mp4` 的文件名。
- 为一批素材分配从指定起点开始的连续编号。
- 根据已成功下载数量计算收藏模式剩余额度。
- 下载成功 key 与失败记录的区分。
- 已取消收藏记录不再进入清理队列，失败和 skipped 记录继续进入。

通过临时目录集成测试验证排他写入不会覆盖已有文件。
